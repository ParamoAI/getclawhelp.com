OpenClaw Security Hardening: Protect Your AI

March 24, 2026 · 14 min read

You have an AI assistant running on your own server. It connects to Slack, Telegram, maybe WhatsApp. It has access to your files, your conversations, your business data. The question is: how well is that server locked down?

Self-hosting OpenClaw gives you full control over your data, but that control comes with responsibility. A misconfigured server is an open invitation. A leaked API key can rack up thousands of dollars in charges. An unpatched operating system is a vulnerability waiting to be exploited.

This guide covers everything you need to harden your OpenClaw security from the ground up: SSH lockdown, firewall configuration, secret management, network encryption, monitoring, and ongoing maintenance. Whether you are running OpenClaw on a VPS, a Mac Mini, or a Docker container, these practices apply.

New to OpenClaw? Start with our guide to what OpenClaw is before diving into security.

Why Security Matters for Self-Hosted AI

When you use a managed AI service like ChatGPT or Claude.ai, the provider handles security for you. They manage the servers, patch vulnerabilities, rotate keys, and monitor for intrusions. You trade privacy for convenience.

With OpenClaw, you get the privacy back. Your data never leaves your server (except when sent to the AI model provider for inference). But you also inherit the security obligations. Here is what is at stake:

  • API keys with real dollar signs. Your OpenClaw instance holds API keys for Claude, GPT-4, or other models. If someone steals those keys, they can run up massive bills on your account. A single compromised key can cost hundreds or thousands of dollars before you notice.
  • Channel credentials. Slack bot tokens, Telegram bot tokens, WhatsApp session data: these grant access to your team conversations. A leaked Slack token means someone can read every message in every channel your bot has joined.
  • Business data in memory. OpenClaw stores conversation history, uploaded files, and potentially sensitive business information. Depending on your setup, this could include customer data, financial discussions, or strategic plans.
  • Server access as a pivot point. A compromised OpenClaw server can be used to attack other systems on the same network. If your AI server sits on the same network as your database or internal tools, one breach can cascade.

The good news: hardening an OpenClaw deployment is not complicated. It is a checklist, and this guide walks you through every item.

1. Lock Down SSH Access

SSH is the front door to your server. If you are running OpenClaw on a VPS, SSH is how you manage it remotely. Getting SSH right is the single most impactful security step you can take.

Use SSH Keys, Not Passwords

Password-based SSH login is the number one attack vector for internet-facing servers. Automated bots scan the internet constantly, trying common passwords against SSH servers. Key-based authentication eliminates this attack entirely.

Generate an SSH key pair on your local machine if you do not have one:

ssh-keygen -t ed25519 -C "your-email@example.com"

Copy the public key to your server:

ssh-copy-id -i ~/.ssh/id_ed25519.pub user@your-server-ip

Then disable password authentication in /etc/ssh/sshd_config:

PasswordAuthentication no
PubkeyAuthentication yes
PermitRootLogin no
MaxAuthTries 3

Restart the SSH service:

sudo systemctl restart sshd

Important: Test your key-based login in a separate terminal before closing your current session. If something is misconfigured, you could lock yourself out.

Change the Default SSH Port

Moving SSH off port 22 does not stop a determined attacker, but it eliminates 99% of automated bot traffic. Less noise in your logs means real threats are easier to spot.

# In /etc/ssh/sshd_config
Port 2222  # Pick any unused port above 1024

Remember to update your firewall rules (covered next) and your SSH client config to use the new port.

Add Fail2Ban

Fail2Ban monitors authentication logs and temporarily bans IP addresses that show malicious behavior, like repeated failed login attempts.

sudo apt install fail2ban
sudo systemctl enable fail2ban
sudo systemctl start fail2ban

The default configuration protects SSH out of the box. For custom rules, create a file at /etc/fail2ban/jail.local:

[sshd]
enabled = true
port = 2222
maxretry = 3
bantime = 3600
findtime = 600

This bans any IP that fails 3 login attempts within 10 minutes, for 1 hour.

2. Configure Your Firewall

A firewall controls which network traffic can reach your server. Without one, every service running on your server is exposed to the internet. With one, you allow only the traffic you explicitly need.

UFW on Ubuntu/Debian

UFW (Uncomplicated Firewall) is the simplest way to manage firewall rules on Ubuntu and Debian servers:

# Allow SSH (use your custom port if changed)
sudo ufw allow 2222/tcp

# Allow HTTPS if you serve a webhook
sudo ufw allow 443/tcp

# Enable the firewall
sudo ufw enable

# Check status
sudo ufw status verbose

The key principle: deny by default, allow by exception. UFW blocks all incoming traffic by default once enabled. You only open the ports you need.

What Ports Does OpenClaw Need?

In most configurations, OpenClaw does not need any inbound ports open (beyond SSH for management). Here is why:

  • Slack (Socket Mode) uses outbound WebSocket connections. No inbound ports needed.
  • Telegram uses outbound HTTPS polling. No inbound ports needed.
  • WhatsApp (Baileys) uses outbound WebSocket connections. No inbound ports needed.

The only time you need to open an inbound port (besides SSH) is if you are running a webhook-based integration or exposing a web interface. In that case, open only port 443 and put it behind a reverse proxy with TLS.

macOS Firewall for Mac Mini Setups

If you are running OpenClaw on a Mac Mini, enable the built-in macOS firewall:

  1. Open System Settings and go to Network.
  2. Click Firewall and toggle it on.
  3. Click Options to configure which apps can receive connections.

For more granular control, use pfctl (the macOS packet filter) or install a tool like Little Snitch.

3. Manage Secrets Properly

Your OpenClaw configuration file holds sensitive credentials: AI model API keys, Slack bot tokens, Telegram tokens, and more. How you store and manage these secrets is critical.

Never Commit Secrets to Git

This sounds obvious, but it happens constantly. If your OpenClaw configuration is in a Git repository, make sure the config file with tokens is in .gitignore. Better yet, use environment variables instead of hardcoding secrets in the config file.

# .gitignore
openclaw.json
.env
*.key
*.pem

Use Environment Variables

Store sensitive values in environment variables rather than directly in your configuration file. Create a .env file (which should be in .gitignore):

# .env
ANTHROPIC_API_KEY=sk-ant-...
SLACK_BOT_TOKEN=xoxb-...
SLACK_APP_TOKEN=xapp-...
TELEGRAM_BOT_TOKEN=123456:ABC...

Reference these in your OpenClaw configuration using the ${ENV_VAR} syntax that OpenClaw supports.

Set Proper File Permissions

Your configuration file and any .env files should be readable only by the user running OpenClaw:

chmod 600 openclaw.json
chmod 600 .env

The 600 permission means only the file owner can read and write it. No other users on the system can access your tokens.

Rotate Keys Regularly

Set a calendar reminder to rotate your API keys every 90 days. The process is straightforward:

  1. Generate a new key in your model provider's dashboard.
  2. Update the key in your OpenClaw configuration or environment variables.
  3. Restart the OpenClaw gateway to pick up the new key.
  4. Verify the new key works by sending a test message.
  5. Revoke the old key in the provider dashboard.

Never revoke the old key before confirming the new one works. You do not want to accidentally take your AI assistant offline.

4. Run OpenClaw as a Non-Root User

Never run OpenClaw as root. If the process is compromised, the attacker gets root access to your entire server. Instead, create a dedicated user:

sudo adduser --system --group --home /opt/openclaw openclaw-user
sudo chown -R openclaw-user:openclaw-user /opt/openclaw

If you are using systemd to manage OpenClaw as a service, set the user in your service file:

[Service]
User=openclaw-user
Group=openclaw-user
WorkingDirectory=/opt/openclaw
ExecStart=/usr/bin/openclaw gateway start

This limits the blast radius. Even if someone finds a way to exploit OpenClaw, they can only access files owned by the openclaw-user account, not the entire system.

5. Keep Everything Updated

Unpatched software is the most common entry point for attackers. A single known vulnerability in your operating system, Node.js runtime, or a dependency can be exploited within hours of public disclosure.

Enable Automatic Security Updates

On Ubuntu/Debian, enable unattended security updates:

sudo apt install unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades

This automatically installs security patches without requiring manual intervention. Non-security updates still require manual review.

Update OpenClaw

Check for OpenClaw updates regularly:

npm update -g openclaw

Subscribe to the OpenClaw GitHub repository or release announcements so you are notified when new versions drop. Security fixes are often released as patch versions.

Update Node.js

OpenClaw runs on Node.js. Keep it updated using a version manager like nvm:

nvm install --lts
nvm alias default lts/*

Stick to LTS (Long-Term Support) releases for stability and security patches.

6. Docker Security for Containerized Deployments

If you are running OpenClaw in Docker, containers add a layer of isolation. But Docker has its own security considerations.

Run Containers as Non-Root

By default, Docker containers run as root inside the container. Override this in your Dockerfile or docker-compose file:

# docker-compose.yml
services:
  openclaw:
    image: openclaw:latest
    user: "1000:1000"
    read_only: true
    security_opt:
      - no-new-privileges:true
    tmpfs:
      - /tmp

Limit Container Resources

Prevent a compromised container from consuming all host resources:

services:
  openclaw:
    deploy:
      resources:
        limits:
          cpus: '2.0'
          memory: 2G

Use Docker Secrets for Credentials

Instead of passing API keys as environment variables (which show up in docker inspect), use Docker secrets:

echo "sk-ant-your-api-key" | docker secret create anthropic_key -

# Reference in docker-compose.yml
services:
  openclaw:
    secrets:
      - anthropic_key
secrets:
  anthropic_key:
    external: true

7. Monitor and Audit

Hardening is not a one-time event. You need ongoing visibility into what is happening on your server.

Review Logs Regularly

Check authentication logs for suspicious activity:

# Failed SSH attempts
sudo grep "Failed password" /var/log/auth.log | tail -20

# Successful logins
sudo grep "Accepted" /var/log/auth.log | tail -20

# OpenClaw gateway logs
openclaw gateway logs

Set Up Basic Monitoring

At minimum, monitor these metrics:

  • Disk usage. A full disk can crash OpenClaw and prevent logging. Set up alerts at 80% capacity.
  • Memory usage. Memory leaks or unexpected spikes can indicate a problem.
  • Failed login attempts. A spike in failed SSH attempts means someone is trying to break in.
  • API usage. Monitor your AI model provider dashboard for unusual usage patterns that could indicate a stolen key.

Simple tools like htop, df -h, and journalctl cover the basics. For more sophisticated monitoring, consider Uptime Kuma (self-hosted) or a hosted monitoring service.

API Key Usage Alerts

Most AI model providers let you set spending limits and alerts:

  • Anthropic: Set a monthly spend limit in the Anthropic Console under Billing.
  • OpenAI: Set a hard usage cap in the OpenAI Platform under Billing.

Always set a spending cap. Even if your server is never compromised, a runaway automation or misconfigured skill can burn through your budget quickly.

8. Network Security

How your OpenClaw server communicates with the outside world matters.

Use a VPN for Remote Management

Instead of exposing SSH to the public internet, consider using a VPN like WireGuard or Tailscale. You connect to the VPN first, then SSH to your server through the private network. This makes your SSH port invisible to the public internet entirely.

# Install Tailscale (one command)
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up

# Now SSH via your Tailscale IP
ssh user@100.x.y.z

Isolate Your OpenClaw Server

If possible, run OpenClaw on a network segment separate from your other infrastructure. On cloud providers like AWS, DigitalOcean, or Hetzner, use a private network (VPC) and place OpenClaw in its own subnet with strict security group rules.

Encrypt All Traffic

All communication between OpenClaw and external services (Slack, Telegram, AI model APIs) already uses TLS encryption. Verify by checking that all URLs in your configuration use https://. Never use plain HTTP for API calls or webhook endpoints.

9. Backup Your Configuration

A security incident can mean losing your configuration, conversation history, and skills. Regular backups let you recover quickly.

  • Back up your config file. Copy openclaw.json (with secrets removed) and your .env file to a secure location. Encrypted cloud storage or an offline USB drive works.
  • Back up your workspace. If you use SOUL.md, AGENTS.md, skills, and memory files, these represent your AI's accumulated knowledge. Back them up to a private Git repository.
  • Test your backups. A backup you have never restored from is not a backup. Periodically test that you can spin up a fresh server from your backups.

10. Security Checklist

Here is a condensed checklist you can work through to harden your OpenClaw deployment. Print it, bookmark it, or save it to your notes:

  • ☐ SSH key authentication enabled, password authentication disabled
  • ☐ Root login disabled over SSH
  • ☐ SSH port changed from default 22
  • ☐ Fail2Ban installed and configured
  • ☐ Firewall enabled with only necessary ports open
  • ☐ OpenClaw running as a non-root user
  • ☐ API keys stored in environment variables, not in code
  • ☐ Config file permissions set to 600
  • ☐ .gitignore excludes all secret files
  • ☐ Automatic OS security updates enabled
  • ☐ Node.js on latest LTS version
  • ☐ OpenClaw on latest version
  • ☐ Spending limits set on AI model provider accounts
  • ☐ Backups configured and tested
  • ☐ Log review scheduled (weekly minimum)
  • ☐ API key rotation scheduled (90-day cycle)

You do not need to complete every item on day one. Prioritize SSH and firewall first, then work through the rest over the next week.

Common Security Mistakes to Avoid

These are the mistakes we see most often when helping teams set up OpenClaw:

  • Running as root. It is convenient but dangerous. Always create a dedicated user.
  • Leaving SSH on port 22 with password auth. This is the equivalent of leaving your front door unlocked. Fix it immediately.
  • Committing API keys to GitHub. Even in a private repo, this is a risk. Use environment variables and .gitignore.
  • No spending limits on model APIs. A stolen API key with no spending cap is a blank check. Set limits from day one.
  • Ignoring updates for months. Security patches exist for a reason. A 5-minute update can prevent a 5-day incident response.
  • Opening all ports "just to get it working." Start with everything closed and open only what you need. Never the reverse.

Frequently Asked Questions

Is OpenClaw secure by default?

OpenClaw follows reasonable defaults, but no software is fully hardened out of the box. Self-hosted means you control the security posture. The steps in this guide bring your deployment from baseline to production-grade by locking down SSH, configuring firewalls, managing secrets properly, and enabling automatic updates.

Do I need a firewall if OpenClaw runs on a home Mac Mini?

Yes. Even on a home network, a firewall prevents unexpected inbound connections. macOS has a built-in firewall you can enable in System Settings under Network. If your Mac Mini is exposed to the internet via port forwarding, a firewall is absolutely critical.

How do I rotate my AI model API keys?

Go to your model provider dashboard (e.g., Anthropic Console or OpenAI Platform), generate a new API key, update your OpenClaw configuration or environment variables with the new key, restart the gateway, then revoke the old key. Do not delete the old key before confirming the new one works.

Should I use Docker for better security?

Docker adds an isolation layer that can improve security. Your OpenClaw process runs inside a container with limited access to the host system. Combined with a non-root user, read-only filesystem mounts, and resource limits, Docker is a solid choice for production deployments. See our Docker setup guide for details.

Can someone intercept messages between OpenClaw and Slack or Telegram?

OpenClaw uses TLS-encrypted connections for all channel integrations. Slack uses Socket Mode over WSS (WebSocket Secure), Telegram uses HTTPS for its Bot API, and WhatsApp uses end-to-end encryption. The weakest link is usually the server itself, which is why hardening the host machine matters.

How often should I update OpenClaw?

Check for updates at least weekly. Security patches can drop at any time. Enable automatic OS-level security updates (unattended-upgrades on Ubuntu/Debian) and subscribe to the OpenClaw release channel so you know when a new version is available. Always test updates on a staging instance first if you run a production workload.
